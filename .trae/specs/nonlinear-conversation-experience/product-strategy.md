# DeepMindMap 非线性对话体验产品策略（v1.2-final）

> 文档版本：v1.2-final
> 更新日期：2026-06-19
> 定位：面向后续 2-3 个月开发的产品方向与功能优先级文档

***

## 一、背景与核心痛点

### 1.1 用户真实痛点

现有通用 AI 对话（DeepSeek、ChatGPT、豆包）是线性的：

* 用户在了解复杂主题时，经常会对回答中的某个细节产生疑问。

* 继续追问细节，会让整个对话偏离最初的主题。

* 等细节聊完后，用户往往已经忘记最初的核心问题或思路。

DeepMindMap 要解决的底层痛点：**让 AI 对话像人的思维一样，有主线、有支线，深入后还能回来，并最终沉淀为一张可复用的知识地图。**

### 1.2 当前产品的问题

通过代码审查与实际体验，项目已经搭建了非线性对话的“容器”：

* 每个节点有独立的对话上下文。

* 子节点可继承父节点历史。

* 支持父子、支持、矛盾、前提、细化、参考、结论、自定义等 8 种关系。

* 支持复合节点、自动布局、多选模式。

* 对话面板中已有 `MindMapThumbnail` 节点导航树（左上角）。

* 已有“提炼结论”按钮和“+ 分支”按钮。

但实测下来，最大的问题是：

| 问题编号 | 问题描述                       | 优先级 | 当前状态                    |
| ---- | -------------------------- | --- | ----------------------- |
| A    | 用户根本不会主动创建分支，仍然把产品当普通聊天工具用 | P0  | 前端有“+ 分支”按钮，但入口隐蔽       |
| B    | 分支操作麻烦，成本高，打断了对话节奏         | P0  | 需要用户手动点击、命名、切换          |
| D    | 支线聊完后，结论没有自然沉淀到主线，知识散落     | P1  | 已有“提炼结论”，但需手动触发且未写入节点摘要 |
| E    | 没有用户行为埋点，无法判断功能是否真正被使用     | P0  | 完全缺失                    |
| C    | 子节点继承太多主线上下文（体验上感觉不明显）     | 低   | 可控，暂不影响                 |

### 1.3 关键洞察

> 项目已经搭好了“非线性的容器”，但没有提供“非线性的默认行为”。
>
> 用户不是不想用分支，而是分支对他们来说是“额外功能”，而不是“聊天的自然下一步”。
>
> **更深一层洞察**：新用户面对空白画布时，不知道该做什么。产品的“啊哈时刻”不是画出一张完美地图，而是在第一次对话中，用户发现自己“自然地深入了一个细节，又能一键回到主线”。

## 三、产品定位

DeepMindMap 的目标场景不局限于“学习”，而是任何需要**结构化深度探索**的场景：

* **学习**：学一个新领域，逐层深入知识点。

* **研究**：读论文、查资料，把零散信息组织成知识网络。

* **产品/项目分析**：拆解问题、梳理方案、评估 trade-off。

* **创意发散**：头脑风暴，从一个点子衍生出多个方向。

* **技术方案设计**：需求分析、架构设计、技术选型。

### 3.1 一句话定位

> **把一次深度 AI 探索，变成一张可管理、可摘要、可沉淀的思维地图。**

### 3.2 与 DeepSeek/ChatGPT 的关系

不试图替代通用 AI 聊天。目标是在“需要深入、需要结构化”的场景下，成为比线性聊天更高效的工具。

### 3.3 差异化壁垒

| 维度    | 线性 AI 聊天    | DeepMindMap  |
| ----- | ----------- | ------------ |
| 主线/支线 | 无，所有内容混在一起  | 清晰，支线不污染主线   |
| 深入细节  | 需要复制粘贴、开新窗口 | 一键分叉，上下文自动继承 |
| 结论沉淀  | 靠用户自己总结     | 自动为节点生成摘要    |
| 复盘复习  | 翻历史记录       | 直接看地图结构      |

## 四、设计原则

### 4.1 默认非线性

用户不需要理解“节点”“分支”这些概念。分支应该是聊天的自然下一步。

### 4.2 低摩擦分叉

从“想深入”到“进入支线”，最多一次点击。

### 4.3 主线始终可见

无论用户在多深的支线上，都要能一眼看到主线在哪里，一键回去。

### 4.4 节点摘要

支线的价值不能只停留在对话里，要自动或半自动地为每个节点提炼摘要，让地图上的每个节点都一目了然。

### 4.5 用数据说话

任何功能上线前必须先有埋点，上线后用数据判断效果，而不是靠感觉。

### 4.6 成本控制

考虑到当前主力模型为 GLM-4-Flash（免费/低成本），所有智能化能力都要避免额外调用模型，优先用 prompt 工程、规则匹配、前端解析实现。

## 五、核心用户旅程

```text
1. 用户输入主题或问题
        ↓
2. AI 给出结构化回答
   并在末尾提供 2-4 个可点击的“延伸方向”
        ↓
3. 用户点击某个延伸方向
   → 自动创建子节点
   → 自动切换到该子节点
   → 自动把对应问题发送给 AI
   → 用户进入子节点对话
        ↓
4. 子节点中 AI 继续回答，并再次提供延伸方向
   用户也可自由输入
        ↓
5. 用户聊完支线
   → 点击“回到主线”或在节点树中点击父节点
   → 系统自动提炼该支线节点的核心内容，写入该节点的“摘要”字段
        ↓
6. 最终得到一张每个节点都有清晰摘要、可继续迭代、可复习、可分享的思维地图
```

### 5.1 新用户冷启动路径

```text
新用户进入
   ↓
自动弹出新手引导（已更新为 8 页）
   ↓
进入默认工作区，看到“地图库”/“模板库”
   ↓
推荐“DeepMindMap 使用指南”模板
   ↓
用户选择模板 → 系统自动复制到工作区并打开
   ↓
在已有节点上和 AI 对话
   ↓
AI 回答 + 延伸方向按钮
   ↓
用户点击延伸方向 → 自动创建子节点并进入
   ↓
啊哈时刻：原来可以这样深入！
   ↓
聊完后点击“回到上一级” → 自动生成节点摘要
   ↓
地图库中保存了第一张有结构的地图
```

### 5.2 老用户回归路径

```text
老用户登录
   ↓
工作区选择页面显示“我的工作区” + “推荐工作区”
   ↓
进入工作区 → 地图库展示所有历史地图
   ↓
搜索或点击最近编辑的地图
   ↓
继续之前的探索
```

## 六、功能模块与优先级

### P0：回答内可点击的“延伸方向”

#### 功能描述

每次 AI 回答后，在消息底部显示 2-4 个可点击的“深入方向”按钮。用户点击后：

1. 自动创建子节点。
2. 自动切换到该子节点。
3. 自动把对应问题发送给 AI。
4. AI 在子节点中给出深入回答。

#### 示例

> AI 回答完“泰勒展开”后，消息底部出现：
>
> 想深入了解：① 泰勒展开的推导 ② 泰勒展开 vs 麦克劳林展开 ③ 常见错误与边界条件
>
> 用户点①，自动创建子节点“泰勒展开的推导”，并发送问题：“请详细解释泰勒展开的推导过程。”

#### 成本策略

* 不需要额外调用模型。

* 利用现有 system prompt 中已有的 `🌱 延伸方向：` 格式要求。

* 前端解析 AI 输出中的固定格式，提取方向列表并渲染成按钮。

* 如果 AI 没有输出该格式，则不显示按钮，避免强依赖。

#### 参考实现位置

* `server/src/config/prompts.ts` 已包含延伸方向格式要求。

* `client/src/components/Chat/ChatPanel.tsx` 中可扩展消息渲染逻辑。

* `client/src/stores/appStore.ts` 中 `createChildNode` 可直接复用。

#### 验收标准

* [ ] 流式响应结束后，若 AI 输出 `🌱 延伸方向：` 格式，则渲染为按钮。

* [ ] 点击按钮后，1 秒内创建子节点并切换过去。

* [ ] 子节点自动发送对应问题并触发 AI 回答。

* [ ] 无延伸方向时不显示空区域。

### P0：用户行为埋点（后台管理集成）

#### 功能描述

用户行为数据统一接入**后台管理系统**的 Dashboard，作为产品迭代的数据基础。不在主服务中单独维护一套简单事件接口。

#### 必须追踪的事件

| 事件                       | 触发时机        | 用于验证      |
| ------------------------ | ----------- | --------- |

### P1：优化现有节点导航树（替代“常驻回到主线”）

#### 问题修正

原策略中提出的“常驻回到主线入口”与现有实现冲突。对话面板左上角已有 `MindMapThumbnail` 节点导航树，可以直接点击父节点返回主线。无需再增加一个独立按钮。

#### 功能描述

在现有 `MindMapThumbnail` 基础上增强，让用户在支线中不迷失：

1. **高亮当前路径**：当前节点及其父节点链用不同颜色高亮。
2. **快捷回到父节点**：在导航树标题栏增加“← 回到上一级”按钮，一键选中父节点。
3. **自动展开当前节点**：进入子节点时，自动在树中展开并滚动到当前节点。
4. **移动端优先展示**：移动端默认展开导航树，让用户清楚自己所在位置。

#### 示例

> 当前在“泰勒展开的推导”子节点中，导航树标题显示：
>
> 思维导图 6 节点　← 回到“泰勒展开”
>
> 点击后选中父节点“泰勒展开”，聊天面板切换回父节点对话。

#### 参考实现位置

* `client/src/components/Chat/MindMapThumbnail.tsx`

* `client/src/stores/appStore.ts` 中 `selectNode`

#### 验收标准

* [ ] 导航树中当前节点到根节点的路径高亮显示。

* [ ] 当前节点非根节点时，标题栏显示“回到 \[父节点标题]”按钮。

* [ ] 点击后选中父节点并切换对话。

* [ ] 进入子节点时，树自动展开并滚动到当前节点。

### P1：输入框智能分叉提示

#### 功能描述

当用户在当前节点输入新问题时，系统判断这个问题是否更像“针对某个子主题的深入”。如果是，则弹出轻提示：

> “你似乎想深入聊‘XXX’，是否创建一个分支？这样不会打乱主线。”

用户点击“创建分支”后，自动创建子节点并带入问题。

#### 判断策略（第一阶段规则匹配，不调用模型）

**A. 明确深入关键词**（出现任意一个即触发提示）：

* 详细解释、深入讲解、展开讲讲、具体说说

* 为什么、怎么回事、什么原因

* 有什么区别、有什么不同、对比

* 举个例子、举个例子说明、能不能举例

* 优缺点、优势和劣势、利弊

* 适用场景、什么时候用、什么情况下

* 实现原理、底层原理、工作原理

* 常见错误、容易踩的坑、注意事项

* 历史背景、发展历程、由来

* 最佳实践、推荐方案、怎么做最好

**B. 语义偏离检测**（满足任意一条即触发）：

* 用户问题中提到的核心概念与当前节点标题的交集较低（例如当前节点是“泰勒展开”，用户问“麦克劳林展开是什么”）。

* 用户连续 2 轮以上追问同一子主题。

**C. 反问确认文案**：

> “这个问题更像是在聊‘麦克劳林展开’，是否创建一个分支？”

#### 产品目标

把“用户主动创建分支”变成“用户被提示后一键确认”，降低分支操作成本。

### P1：支线节点自动摘要（替代“回流/归档”）

#### 问题修正

原策略中提出的“在父节点下生成结论子节点”会让画布多出许多节点，造成视觉干扰。实际上：

* 支线本身就是一个子节点。

* 结论应该属于这个支线节点，而不是再挂一个新的节点。

本版改为：**给每个节点增加一个“摘要”字段，聊完支线后自动生成该节点的摘要。**

#### 功能描述

当用户结束支线探索后，系统自动提炼该支线节点对话的核心内容，写入该节点的 `summary` 字段：

* 摘要内容：1-3 句核心结论 + 3-5 个要点 bullet

* 存储位置：节点自身的 `summary` 字段（不新增节点）

* 展示方式：

  * 画布中：节点卡片下方显示摘要预览（可折叠）

  * 节点导航树中：悬停显示摘要

  * 聊天面板中：节点信息头部下方展示摘要

这样用户看到的是：

* 父节点：原始主题

* 子节点 A：深入探索的分支，附带清晰摘要

画布不增加额外节点，地图依然清晰。

#### 触发条件

| 触发方式    | 说明                                       |
| ------- | ---------------------------------------- |
| 手动触发    | 用户点击“生成摘要”按钮（替代原“提炼结论”按钮）                |
| 返回主线时触发 | 用户在节点树中点击父节点或“回到上一级”时，若该分支有新增对话，提示是否生成摘要 |
| 自动触发    | 子节点对话超过 4-6 轮，且 3 分钟内无新消息，提示用户是否生成摘要     |

#### 结束支线探索的判断标准

系统通过以下信号判断用户可能已结束支线探索：

1. **显式信号**：用户点击“回到上一级”或“生成摘要”。
2. **时间信号**：子节点最后一条消息超过 3 分钟未继续输入。
3. **轮数信号**：子节点对话轮数达到 4-6 轮，且最后一条是用户表示感谢、总结或“明白了”等结束语。
4. **语义信号**：用户输入包含“总结”“明白了”“先这样”“回来”“回到主线”等结束语。

> **宁可少生成，不要误生成**。误生成摘要会污染节点信息；漏生成用户可以手动触发。

#### 成本策略

* 仅在触发时调用一次模型进行摘要提炼。

* 结论提炼 prompt 已存在：`server/src/config/prompts.ts` 中的 `CONCLUSION_EXTRACTION_PROMPT`，可调整为生成节点摘要。

* 前端无需新增节点创建逻辑，只需扩展节点数据模型和展示层。

#### 涉及的数据模型变更

* `NodeData` 增加可选字段 `summary?: string`。

* 后端节点模型同步增加 `summary` 字段。

* 现有节点无摘要时，显示“暂无摘要”。

#### 验收标准

* [ ] 手动点击“生成摘要”后，当前节点的 `summary` 字段被更新。

* [ ] 返回主线时若子节点有新增对话，提示用户是否生成摘要。

* [ ] 摘要内容包含核心结论和要点 bullet。

* [ ] 画布中节点卡片可折叠显示摘要预览。

* [ ] 误触发率低，复杂对话不自动生成摘要。

### P1：后台 AI 模型 / API 池管理

#### 功能描述

当前系统已支持内置 AI 服务和用户自定义 API Key。为了更灵活地控制成本、尝试新模型、优化不同任务的效果，需要在后台管理系统中增加 **AI 模型 / API 池** 管理能力。

管理员可以在后台：

1. **添加/编辑/删除内置 AI 模型配置**

   * 模型名称（如“GLM-4-Flash”“DeepSeek-V3”“GPT-4o-mini”）

   * 服务提供商（智谱、DeepSeek、OpenAI、Azure 等）

   * API Key / 访问令牌

   * Base URL（支持代理或私有化部署）

   * 模型 ID（实际请求的 model 参数）

   * 默认参数：temperature、max\_tokens、top\_p 等

   * 启用/禁用状态

2. **设置默认模型和优先级**

   * 指定一个默认模型供所有用户使用。

   * 设置多个模型的优先级，形成主备链路（如 A 失败自动fallback到 B）。

3. **按任务类型路由模型（可选）**

   * 普通对话：使用便宜模型（如 GLM-4-Flash）

   * 延伸方向生成：使用同一模型即可（已在回答中输出）

   * 节点摘要/结论提炼：使用质量更高的模型

   * 地图优先模式生成大纲：使用质量较高的模型

4. **用量与成本监控**

   * 每个模型的调用次数、token 消耗、失败率。

   * 在 admin Dashboard 中展示，便于管理员评估性价比。

#### 示例

> 管理员在后台看到当前内置模型列表：
>
> | 模型          | 提供商      | 状态 | 默认 | 用途    |
> | ----------- | -------- | -- | -- | ----- |
> | GLM-4-Flash | 智谱       | 启用 | 是  | 默认对话  |
> | DeepSeek-V3 | DeepSeek | 启用 | 否  | 摘要/大纲 |
> | GPT-4o-mini | OpenAI   | 禁用 | 否  | 备用    |
>
> 管理员可以点击“添加模型”，填入新模型的 API Key 和参数，立即生效。

#### 产品目标

* **降低成本**：根据任务选择最便宜够用的模型。

* **提升质量**：对不同任务使用更合适的模型。

* **快速试错**：看到新模型或便宜的模型时，可以快速接入测试效果。

* **保障稳定性**：主模型失败时自动 fallback 到备用模型。

#### 与用户的关系

* 默认情况下用户无需感知模型切换，由后端统一调度。

* 可选：在设置中允许用户从“可用内置模型”列表中选择自己喜欢的模型（高级用户）。

* 用户自填 API Key 的原有能力保留，优先级高于内置模型池。

#### 与 P0 的关系

延伸方向、节点摘要、地图优先模式都依赖 AI 输出质量。API 池管理是这些功能的基础设施：

* 如果 GLM-4-Flash 输出的延伸方向质量不稳定，可以切换到 DeepSeek-V3。

* 如果节点摘要需要更强的总结能力，可以单独配置用更强的模型。

#### 参考实现位置

* Admin 页面：`admin/client/src/pages/` 新增 `AIModelsPage.tsx`

* Admin API：`admin/server/src/routes/` 新增 `aiModels.ts`

* 主服务：`server/src/services/aiService.ts` 读取模型池配置

* 配置存储：可在数据库中新增 `ai_model_configs` 表/集合，或复用现有配置表

#### 验收标准

* [ ] 后台可增删改查内置 AI 模型配置。

* [ ] 支持设置默认模型和优先级/fallback 链路。

* [ ] 支持按任务类型（对话/摘要/大纲）配置不同模型。

* [ ] Admin Dashboard 展示各模型调用量、失败率、token 消耗。

* [ ] 切换模型后无需重启服务即可生效（或最小重启成本）。

* [ ] 用户自填 API Key 的能力不受影响。

### P2：地图优先模式

#### 功能描述

针对某些复杂问题，AI 第一步不是直接回答，而是先给出一个结构化大纲，生成到画布上：

> 用户问：“如何设计一个高并发系统？”
> AI 生成根节点和多个分支：架构选型、缓存策略、数据库设计、限流降级、监控告警……

#### 适用场景

* 用户提出的是一个宽泛主题，而不是具体问题。

* 学习、研究、方案设计类场景。

#### 与 P0 的关系

地图优先模式可以作为“新对话”的一种入口形态。用户在创建新主题时，可选择：

* “直接回答我”（现有模式）

* “帮我先展开成一张地图”（地图优先模式）

#### 实现建议

* 复用现有 `expand_node` AI 工具，让 AI 自动创建多个子节点。

* 在空状态或根节点输入宽泛问题时，提示“是否先展开成地图？”

## 七、成本控制策略

考虑到当前主力使用 GLM-4-Flash，所有设计都要避免额外模型调用：

| 功能      | 成本策略                                |
| ------- | ----------------------------------- |
| 延伸方向按钮  | 由 AI 在主回答中一并输出，前端解析，不额外调用           |
| 输入框分叉提示 | 规则匹配/关键词判断，不调用模型                    |
| 自动创建子节点 | 复用现有 `createChildNode`，不调用模型        |
| 节点摘要    | 仅在用户触发时调用一次模型，且可通过 API 池路由到性价比更高的模型 |
| 地图优先模式  | 由 AI 一次生成完整大纲，不逐节点多次调用              |
| 埋点      | 仅记录事件，不调用模型                         |
| API 池管理 | 通过后台配置选择 cheapest-fit 模型，无需额外模型调用   |

## 九、风险与待验证假设

### 9.1 产品风险

* **用户是否接受 AI 每次回答都给出延伸方向？** 可能觉得烦，需要允许关闭。

* **地图优先模式是否会让简单问题变重？** 需要判断问题复杂度再决定是否启用。

* **分支过多导致地图混乱？** 需要配合折叠、自动布局、节点摘要一起优化。

* **移动端画布体验是否能承载复杂地图？** 需要优先采用“聊天优先”视图。

### 9.2 技术风险

* **GLM-4-Flash 能否稳定输出结构化格式？** 需要通过 prompt 工程和 fallback 处理。

* **输入框分叉判断准确率？** 第一阶段用规则，宁可少提示，不要误提示。

* **节点摘要误触发？** 需要保守的触发策略和明确的手动覆盖入口。

* **多模型切换后输出质量不稳定？** 不同模型对 prompt 的响应差异大，需要为每个模型微调 prompt 或统一封装。

## 十、后续开发优先级建议

### 第一阶段：核心闭环 + 数据基础（2-3 周）

## 十一、验证方式

* 功能完成后，在测试环境中模拟一个复杂主题对话，验证：

  * 是否能一键从回答进入支线。
